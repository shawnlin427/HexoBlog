本期景点： 纳木错+念青唐古拉峰

纳木措，位于西藏自治区中部，是西藏第二大湖泊，也是中国第三大的咸水湖。湖面海拔4718米，形状近似长方形，东西长70多千米，南北宽30多千米，面积约1961km²。位置：30°42′29.89″N， 90°35′46.34。

在从拉萨飞向纳木错的中途，我看到飞行路线西边有一座突出的山峰，对照地图后，发现居然就是赫赫有名的念青唐古拉峰。模拟飞行的乐趣，莫过于此。

模拟飞行

远方的念青唐古拉峰

广阔的纳木错

景点实拍

飞行见闻

  • 从拉萨出发去纳木错，中途看到了布达拉宫。拉萨的周边都是高山，和南方那种被山包围的感觉不同，青藏高原的山都是荒芜的，给人很寂寥的感觉。国内的Bing卫星地图也没有更新，拉萨市区内都是自动生成的盒子楼，感觉更荒凉了。
  • 由于海拔太高，一般的螺旋桨飞机升力不足，在前往纳木错的中途撞在唐古拉山脉上了。最后使用了TBM930飞机。再往北飞一点，就会发现拉萨的地理位置已经相对很好了。难以想象，古时候西藏人民是如何生存的。
  • 从天上看纳木错，没有那么漂亮，只是单纯的大。但是如果站在相对念青唐古拉峰的对岸眺望，那种震撼是无以言表的。

ros2_control

发表于 2022-01-03 更新于 2025-03-27 分类于 robotics ， ROS

为什么我们需要ros/ros2_control？

事实上我们不需要ros_control也可以通过ros完成对机器人的操控。确实如此，因此，每个人都认为他们的机器人都是特别的。我们需要对每一个新的机器人重写控制器，控制框架。这些工作是非常复杂的，特别涉及到一些稍微复杂一点的控制，比如PID，低通滤波，自己实现的效果可能还不是很好。最后我们获得了一堆需要经常维护的代码，俗称屎山。更不要提，很多人其实并不精通控制理论。

那么就有人问了，ros_control，能帮我完成控制吗？他是不是内置了算法？他为什么可以无视控制器的种类，气动，电机，都可以用它吗？

ros_control还提供实时安全的通信。我们自己开发控制框架的时候，也遵循一般的控制原则，就是感知，计算，反馈，但是感知是有固定的刷新频率的，因此实际上是软实时。对于不那么精细的控制来说，这不是大问题，但是对于一些平衡机器人来说，软实时通信的安全性是非常重要的。

ros_control的历史

ros_control的主创团队在2009年开发他们的pr2机器人时就发现，实际上机器人各个关节的控制器本质上是相同的，只是需要参数上的调整，于是开发了一套针对pr2机器人的框架。在2012年，一些研究人员开始尝试将这个框架变得更加泛化，于是结合ros，ros_control问世了。ros_control成功地被部署到多种不同的机器人上，例如双足机器人，轮式机器人。到了2017年，ros2发布，于是ros2_control也在2021年发布了，虽然名字类似，但是是一次很大的返工，适应ros2的新特性，并且修正一些ros_control的不足。

ros2_control的改进点

  • Joint interfaces使用string type，属性不再局限于position， velocity， effort
  • 清晰的代码架构（和使用者关系不大）
  • 默认的ros2_control_node，不用每次都重新写控制node了
  • 清晰的Actuator Sensor System分类
  • 生命周期管理的hw interfaces（这啥意思啊？？）

ros2_control的架构

如何使用ros2_control

  • 在URDF中加入<ros_control>-tag。
  • 部署hw interface，包括Actuator、Sensor、System
  • 调整controller的参数

其中，hw interface是针对传感器，执行器，以及他们组合的接口。什么意思呢？比如说，接受传感器的信息，或者向执行器发布命令，或者与一个集成了编码器的电机交互。

https://jeffzzq.medium.com/designing-a-ros2-robot-7c31a62c535a

https://robotisim.com/2021/07/17/ros2-robotic-arm-bazu-p3-controllers-for-joints-trajectory/

但是很遗憾，目前ros2_control只支持C++的Interface。所以说，如果只会用Python来开发ROS2的话，还有不小的障碍。

ros2_gazebo_pkg的spawn用法

发表于 2022-01-03 更新于 2025-03-27 分类于 robotics ， ROS

spawn用于在Gazebo世界中生成一个实体。为了达成这一目的，有以下几种方法：

使用ROS2 service

命令行的方法

First, run Gazebo with the plugin:

gazebo –verbose -s libgazebo_ros_factory.so

Spawn SDF models

Now call the spawn service from the command line, for example, to spawn a static SDF model:

ros2 service call /spawn_entity ‘gazebo_msgs/SpawnEntity’ ‘{name: “sdf_ball”, xml: “<sdf version=\“1.5\“><model name=\“will_be_ignored\“>true<link name=\“link\“><visual name=\“visual\“>1.0“}’

Spawn SDF lights

To spawn an SDF light, try for example:

ros2 service call /spawn_entity ‘gazebo_msgs/SpawnEntity’ ‘{name: “bulb”, xml: “<sdf version=\“1.5\“><light name=\“will_be_ignored\“ type=\“directional\“>“}’

Spawn URDF robot

Now try this URDF robot:

ros2 service call /spawn_entity ‘gazebo_msgs/SpawnEntity’ ‘{name: “urdf_ball”, xml: “<robot name=\“will_be_ignored\“><link name=\“link\“><sphere radius=\“1.0\“/><mass value=\“1\“/><inertia ixx=\“1\“ ixy=\“0.0\“ ixz=\“0.0\“ iyy=\“1\“ iyz=\“0.0\“ izz=\“1\“/>“}’

Python调用的方法

The above methods is kind of duplicated with automated test. rclpy also provide with api to call the service through Python:

import sys
import rclpy
from gazebo_msgs.srv import SpawnEntity

def request_spawn(xml: str):
rclpy.init()
node = rclpy.create_node(‘spawn_entity’)
client = node.create_client(SpawnEntity, ‘spawn_entity’)
if not client.service_is_ready():
client.wait_for_service()
request = SpawnEntity.Request()
request.xml = xml
future = client.call_async(request)
rclpy.spin_until_future_complete(node, future)
if future.result() is not None:
print(‘response: %r’ % future.result())
else:
raise RuntimeError(‘exception while calling service: %r’ % future.exception())
node.destroy_node()
rclpy.shutdown()

if len(sys.argv) < 2:
print(‘usage: ros2 run my_package my_node.py – example.urdf’)
sys.exit(1)

f = open(sys.argv[1], ‘r’)
request_spawn(f.read())

使用ROS2 run

By using ROS service we can create entity after Gazebo is running. However, sometimes we hope the robot model is placed in the Gazebo as it is launched. To do so, we need to use ROS run and launch.

通过topic的spawn

To spawn from a ros topic, first we need to launch a ros node to publish robot state, ros2 provide the robot_state_publisher package to do so. It accept the urdf as input, and publish the robot_state.

After that, launch the Gazebo with:

ros2 launch gazebo_ros gazebo.launch.py

Then use spawn_entity to generate a robot model.

ros2 run gazebo_ros spawn_entity.py -topic /robot_description -entity robot_name

Obviously, you can do above two actions in a single launch file.

通过sdf/urdf的spawn

It pretty similar with the topic-based methods. All you need to change is to use -file argument instead of -topic, and input the file path of your sdf/urdf.

集成lauchfile

# Gazebo launch
gazebo = IncludeLaunchDescription(

        os.path.join(pkg\_gazebo\_ros, 'launch', 'gazebo.launch.py'),

这个msg的构造出奇的简单！下一步，我们就可以写一个脚本来控制这些关节了。

from geometry_msgs.msg import TransformStamped
from sensor_msgs.msg import Range
from sensor_msgs.msg import Imu
import rclpy
from rclpy.node import Node
import math
from math import pi
import time
import message_filters
import tf_transformations
from sensor_msgs.msg import JointState

class CranebotJointStatePublisher(Node):

def \_\_init\_\_(self):
    super().\_\_init\_\_('cranebot\_joint\_state\_publisher')
    self.imu\_sub = message\_filters.Subscriber(self, Imu, '/cranebot/imu')
    self.tof1\_sub = message\_filters.Subscriber(self, Range, '/cranebot/tof1')
    self.tof2\_sub = message\_filters.Subscriber(self, Range, '/cranebot/tof2')
    self.pub = self.create\_publisher(JointState, '/joint\_states', 10)
    ts = message\_filters.TimeSynchronizer(\[self.imu\_sub, self.tof1\_sub, self.tof2\_sub\], 10)
    ts.registerCallback(self.handle\_cranebot\_pose)
    self.hook\_distance\_init = 0.2


def handle\_cranebot\_pose(self, imu, tof1, tof2):
    #self.get\_logger().info('tof1'+str(tof1.range)+'tof2'+str(tof2.range))
    #self.get\_logger().info('hello')
    ## Tof is used to measure the movement of the cable wire and the arm
    degree = pi/180
    self.arm\_move = float(tof1.range)
    self.wire\_move = float(tof2.range) - self.arm\_move/4
    self.rotate\_move = imu.angular\_velocity.z
    self.raise\_move = imu.angular\_velocity.y
    
    joint\_state = JointState()
    time\_stamp = self.get\_clock().now().to\_msg()
    joint\_state.header.stamp = time\_stamp
    joint\_state.name = \['rotate\_joint', 'raise\_joint', 'extend\_joint', 'pulley\_joint', 'lift\_joint', 'left\_front\_wheel\_joint', 'right\_front\_wheel\_joint', 'left\_back\_wheel\_joint', 'right\_back\_wheel\_joint'\]
    joint\_state.position = \[self.rotate\_move, self.raise\_move, self.arm\_move, -self.raise\_move, self.wire\_move, 0.0, 0.0, 0.0, 0.0\]
    self.pub.publish(joint\_state)

def main():
rclpy.init()
node = CranebotJointStatePublisher()
rclpy.spin(node)
rclpy.shutdown()

if __name__ == ‘__main__‘:
main()

我们只需要定义JointState的header.stamp、name和position，就可以发布topic了。

这样比我们手撸tf、urdf要容易得多！
